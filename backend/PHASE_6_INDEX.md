## Key Concepts

### Trace ID
A unique identifier that flows through the entire request lifecycle:
- Generated by `traceIdMiddleware` for each HTTP request
- Propagated through events, jobs, and workers
- Included in all structured logs
- Returned in response headers

### Structured Logging
JSON-formatted logs with consistent structure:
- Timestamp (ISO format)
- Log level (INFO, ERROR, WARN, DEBUG)
- Message
- Trace ID (for correlation)
- Event ID (for tracking)
- Contextual metadata

### Metrics
System-wide metrics tracked across all components:
- Event counts (emitted, received, failed)
- Job statistics (queued, started, completed, failed)
- Request metrics (method, path, status, duration)
- Listener execution times
- Error counts by type
- Percentile calculations (p50, p95, p99)

### Health Checks
Real-time monitoring of system components:
- Redis connection and queue status
- BullMQ queue health
- Database connection
- Overall system status (healthy, degraded, unhealthy)

## Trace ID Flow

```
HTTP Request
    ↓
traceIdMiddleware
    ├─ Generate/extract trace ID
    ├─ Attach to req.traceId
    └─ Set X-Trace-ID header
    ↓
Route Handler
    ├─ Access req.traceId
    ├─ Emit event with trace ID
    └─ Queue job with trace ID
    ↓
Event Listeners
    ├─ Local listener with trace ID
    └─ Redis listener with trace ID
    ↓
Worker Process
    ├─ Process job with trace ID
    ├─ Emit events with trace ID
    └─ Log with trace ID
    ↓
Response
    └─ Include X-Trace-ID header
```
